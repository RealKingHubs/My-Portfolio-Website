Zero-Secret CI/CD Pipeline for AWS ECSC

I designed and implemented a secure production CI/CD pipeline that deploys a containerized Node.js backend service to AWS ECS Fargate using GitHub Actions with OpenID Connect (OIDC) authentication.

The architecture eliminates long-lived AWS credentials by using temporary IAM roles issued through GitHub’s OIDC federation, creating a zero-secret deployment pipeline. Every code push automatically builds a Docker image, pushes it to Amazon ECR, updates the ECS task definition, and performs a rolling deployment with minimal downtime.

The system also integrates Amazon CloudWatch logging and ECS self-healing, ensuring operational visibility and automatic container recovery in case of failure.

Key Features

  • Zero stored credentials using GitHub OIDC authentication
  • Automated CI/CD pipeline triggered on every push to main
  • Dockerized Node.js backend service
  • Secure container registry using Amazon ECR with image scanning
  • Serverless container orchestration with AWS ECS Fargate
  • Rolling deployments with traceable image tags (Git commit SHA)
  • Self-healing containers automatically restarted by ECS
  • Centralized logging with Amazon CloudWatch

Terraform AWS VPC — Production Infrastructure

Production-grade AWS infrastructure provisioned entirely with Terraform. Built a multi-AZ VPC with public and private subnet tiers, redundant NAT Gateways per Availability Zone, an Application Load Balancer routing HTTP traffic to a private Auto Scaling Group of Apache web servers, and a Bastion Host as the single SSH entry point. Security groups enforce least-privilege access through chaining rather than open CIDR rules. IMDSv2 is enforced on all instances. Remote state is stored in S3 with versioning, AES256 encryption, and DynamoDB locking to prevent concurrent state corruption.

  • Multi-AZ high availability with one NAT Gateway per AZ
  • Private web tier — zero direct internet exposure to web servers
  • Auto Scaling Group with ELB health checks and zero-downtime rolling refresh
  • Least-privilege security via security group chaining
  • IMDSv2 enforced — SSRF protection on all EC2 instances
  • Remote state on S3 with DynamoDB locking and versioning

Secure AWS Infrastructure with Bastion Host and Ansible Automation

Secure AWS architecture with private EC2 instances, Bastion host access, automated Ansible configuration, and load-balanced high availability.

I designed and deployed a secure, highly available web application infrastructure on AWS that isolates backend servers from the public internet while maintaining controlled administrative access. The infrastructure uses private subnets for application servers, a Bastion Host for secure SSH access, and a Load Balancer to distribute traffic across multiple instances, ensuring both security and reliability. All server configurations were automated using Ansible, eliminating manual setup and ensuring consistent deployments.

Key Features

  • Secure network architecture using public and private subnets.
  • Bastion Host (Jump Server) for controlled SSH access to private instances.
  • High availability using a Load Balancer across multiple Availability Zones.
  • Automated configuration management with Ansible playbooks.
  • Infrastructure version control using Git.
  • Deployment validation using a Bash pre-flight script.
  • Dynamic web content showing server hostname and private IP to confirm load balancing

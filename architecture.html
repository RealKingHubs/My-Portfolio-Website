Scalable & Secure AWS Architecture

This architecture demonstrates my Highly Available Web Application Provissioned Using Terraform deployed across two Availability Zones (AZ-1 & AZ-2) within a VPC (10.0.0.0/16).

1. Connectivity & Traffic Management

Inbound Traffic (ALB)

Users connect via the Internet Gateway to an Application Load Balancer (web-alb). It listens on HTTP:80 and spans both AZs to distribute traffic to healthy web servers.

Secure Administration

A Bastion Host in the public subnet serves as a secure "jump server." It allows SSH access to private instances only from a specific administrator's IP address.

2. Tiered Network Segmentation

  • Public Subnets (10.0.1.0/24 & 10.0.2.0/24): These house the NAT Gateways and Elastic IPs. The NAT Gateways allow private instances to reach the internet for updates while blocking unsolicited inbound connections.
  • Private Subnets (10.0.3.0/24 & 10.0.4.0/24): These contain the Apache Web Servers. These instances have no public IPs, ensuring they are isolated from direct internet exposure.

3. High Availability & Scaling

  • Auto Scaling Group (web-asg): Automatically manages the server fleet. It is configured with a minimum of 2 and a maximum of 5 instances, performing health checks via the ELB.
  • Target Group (web-tg): Groups the web instances for the Load Balancer. It monitors health by looking for HTTP 200-399 response codes.
  • Launch Templates: Standardize instance deployment using IMDSv2 with a hop limit of 1 to enhance security against SSRF attacks.

Routing Summary

Public Route Tables: 0.0.0.0/0 → Internet Gateway (IGW)
Private Route Tables: 0.0.0.0/0 → NAT Gateway (NAT GW)

Pipeline Breakdown

This architecture describes my implementation of a CI/CD (Continuous Integration/Continuous Deployment) Pipeline that automates the process of taking code from a developer and deploying it into a secure, scalable AWS environment.

1. GitHub Actions (The Automation Engine)

  • Checkout Code: Pulls the latest code from the repository.
  • OIDC Auth to AWS: Securely connects to AWS using OpenID Connect (OIDC) instead of storing long-term secrets.
  • Docker Build: Packages the application into a Docker image.
  • Push to ECR: Uploads the new image to Amazon ECR (a private storage for Docker images).
  • Update Task Def & Deploy: Updates the instructions for how the app should run and tells AWS ECS to start the new version.

2. AWS Cloud (The Infrastructure)

  • Amazon ECR: A private registry that holds your application images.
  • ECS Cluster (Fargate): This is where the app actually runs. Fargate is "serverless," meaning AWS manages the underlying servers for you.
  • Self-Healing: If a container crashes, ECS automatically detects it and starts a new one to keep the app running.
  • CloudWatch: Collects logs and metrics so you can see how your app is performing and troubleshoot errors.

Multi-AZ Secure Web Architecture

This diagram illustrates my Highly Available and Secure Web Application deployed across two Availability Zones (eu-west-1a & eu-west-1b) in the Ireland region.

1. External Connectivity

Public Traffic (Users)

Traffic flows from the Internet through the Internet Gateway to a Classic Load Balancer. This ensures high availability by distributing requests evenly.

Admin Management

Administrators connect via an SSH Tunnel. They must "jump" through a Bastion Host in the public subnet to reach the private servers.

2. Network Segmentation

  • Public Subnets: Contain the Bastion Host (for management) and the NAT Gateway. The NAT Gateway allows private servers to download updates without being exposed to the internet.
  • Private Subnets: Secure zones hosting Web-Server-1 and Web-Server-2. These instances have Private IPs only and cannot be reached directly from the outside world.

3. Key Infrastructure Components

  • Classic Load Balancer (CLB): Sits in the public layer to receive HTTP traffic and forward it to the private web servers.
  • Multi-AZ Redundancy: The architecture is mirrored across two zones. If one zone fails, the other continues to serve traffic, preventing downtime.
  • Security Groups: Acts as virtual firewalls, restricting traffic so only the Load Balancer can talk to the Web Servers on specific ports.

Network Traffic Legend

Solid Blue: Public Web Traffic (HTTP/80)
Orange Dotted: Secure Management Tunnel (SSH/22)
Green Dotted: NAT Gateway (Outbound Traffic)
